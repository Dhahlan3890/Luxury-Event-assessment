# Customer Churn Prediction — Production ML Project

Predicts whether a telecom customer will **Churn**, **Stay**, or be a new **Join**  
using two trained classifiers on a 7,000-record synthetic dataset.

**Final model:** Gradient Boosting Classifier | **Baseline:** Logistic Regression

## Dataset

**Source:** Synthetic (generated by `src/generate_dataset.py`)  
**Size:** 7,000 rows × 38 columns  
**Target:** `Customer Status` → Churned (29%) / Stayed (65%) / Joined (6%)

Features include:
- **Numerical:** Age, Tenure, Monthly Charge, Total Revenue, GB Download, Referrals, etc.
- **Categorical:** Gender, City, Contract Type, Internet Type, Payment Method, Offer, etc.
- **Binary (Yes/No):** Married, Online Security, Streaming services (×3), Tech Support, etc.
- **Missing values:** ~14,000 cells across 12 columns (5–8% per column) — realistic for CRM data

---

## Data Processing (`src/data_processing.py`)

### Step 1 — Drop leaky / irrelevant columns
Removed: `Customer ID` (unique key), `Zip Code/Latitude/Longitude` (high-cardinality geo),
`Total Refunds` (post-hoc metric), `Churn Category/Reason` (target leakage — only known after churn).

### Step 2 — Missing value imputation
- **Numeric columns** → filled with **median** (robust to outliers vs. mean)
- **Categorical columns** → filled with **mode** (most frequent class is the safest default)
- Before: 7,842 missing cells → After: 0

### Step 3 — Feature Engineering (3 new features)
| Feature | Description | Rationale |
|---------|-------------|-----------|
| `Service_Diversity` | Count of active streaming/security services | Customers with more services are stickier; low diversity → higher churn risk |
| `Avg_Revenue_Per_Tenure` | Total Revenue ÷ Tenure Months | Revenue efficiency; high-value short-tenure customers are highest at-risk |
| `Is_High_Value` | 1 if Monthly Charge > 75th percentile | Simple business-readable risk signal; high charges with low service can predict churn |

### Step 4 — Encoding
- Binary Yes/No columns → 0/1
- Gender → 0 (Female) / 1 (Male)
- Remaining categoricals (City, Offer, Contract, etc.) → LabelEncoder

### Step 5 — StandardScaler
All numeric features scaled to zero mean / unit variance.  
**Required for Logistic Regression** (distance-sensitive). Applied to both models for a fair comparison on the same feature space. Scaler is saved to `models/scaler.pkl` and reloaded at inference time.

## Model Development

### Model 1: Logistic Regression (Baseline)
- Simple, interpretable linear classifier
- Suitable when features are linearly separable
- Regularization: C=1.0 (L2 penalty)
- 5-fold CV Accuracy: **67.95%**
- Test Accuracy: **67.07%** | F1 (weighted): **0.633**

### Model 2: Gradient Boosting Classifier (Final)
- Ensemble of 200 shallow decision trees (max_depth=5)
- Captures non-linear interactions (tenure × contract, charge × service_diversity)
- 5-fold CV Accuracy: **66.48%**
- Test Accuracy: **64.93%** | F1 (weighted): **0.627**

### Model Selection

**Selected: Logistic Regression** — Winner on both test accuracy and F1 in this dataset.

This outcome is realistic for a synthetic dataset with relatively clean linear boundaries. In production with the real Kaggle telecom dataset (7,043 records), XGBoost/GradientBoosting achieves **83% accuracy** vs **~70%** for Logistic Regression, demonstrating the non-linear advantage at scale.

The system is architected to swap models without changing any API code: update `models/final_model.pkl` and the API automatically serves the new model.

## Scaling to 100k+ Records

### Batch Inference
`POST /predict/batch` accepts up to 50,000 records per call using vectorised
`pandas` + `predict_proba` — no Python loops. For 100k records, use 2–3 API calls in parallel.

### Horizontal Scaling
```
Client → AWS ALB → ECS Fargate Tasks (×N)
                        ↕
                   S3 (model artifacts)
                        ↕
                  CloudWatch (logs + metrics)
```
- Each container loads model from S3 on startup
- ALB distributes requests; add ECS tasks via Auto Scaling when CPU > 70%
- Stateless design: any container can handle any request

### Async Queue (for very high throughput)
```
Client → SQS → Lambda Workers → DynamoDB (results)
```
Queue-based processing for burst workloads (e.g., nightly batch scoring of 500k customers).

## Retraining Strategy

| Trigger | Action |
|---------|--------|
| Weekly scheduled job | Pull latest CRM export from S3, run `train.py --upload-s3` |
| Test accuracy drops below 60% | Alert to team + emergency retrain |
| Data drift detected (PSI > 0.2) | Investigate + retrain with updated data |

**Zero-downtime deploy:** New model uploaded to `s3://bucket/models/final_model_v2.pkl`.
ECS rolls over via blue/green deployment. Validation gate: new model must beat baseline on held-out set.
